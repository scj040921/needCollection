## 检索后处理的详细机制

### 召回策略
- **Top‑K选择**：从向量数据库中取出相似度最高的 K 个文档块。K 值通常为 3~10，需平衡上下文长度和信息覆盖度。
- **相似度阈值过滤**：设置一个最低相似度（如 0.7），低于该值的文档块直接丢弃，可减少噪声。
- **最大文档长度限制**：对召回的多个文档块拼接后截断到模型允许的最大 Token 数，防止超长输入导致错误或成本过高。

### 重排（Rerank）
- **原理**：先用 Embedding 做快速召回，再使用更精确的 Cross‑encoder 对候选文档与问题进行深度语义匹配，重新打分排序。
- **常用模型**：
  - `bge-reranker-large`，`Cohere Rerank`，`Jina Reranker` 等。
- **效果**：显著提升最终生成答案的相关性，尤其适用于召回文档较多的情况（如 Top‑20 召回后重排到 Top‑3）。

### 融合排序（Fusion）
- **Reciprocal Rank Fusion（RRF）**：将多个检索结果（如稠密向量检索、BM25）的排名取倒数求和，重新排序。
  $$ \text{score}(d) = \sum_{i=1}^{n} \frac{1}{k + \text{rank}_i(d)} $$
  其中 \(k\) 为常数（常设为 60），\(rank_i(d)\) 是文档 \(d\) 在第 \(i\) 个检索结果列表中的排名。
- **加权分数融合**：将不同来源的相似度分数归一化后加权求和，如 \(score = 0.7 \times sim_{dense} + 0.3 \times sim_{bm25}\)。

## 混合检索与 BM25

### BM25 算法
- **原理**：基于词频（TF）和逆文档频率（IDF）的概率检索模型，考虑文档长度归一化，能有效衡量关键词匹配度。
- **与 TF‑IDF 的区别**：BM25 在 TF 上采用饱和函数（不会因词频高而过快增长），且引入文档长度惩罚参数 \(b\) 和词频饱和度参数 \(k_1\)。
- **工程实现**：通常使用 Elasticsearch、Lucene 或 Python 的 `rank-bm25` 包。

### 混合检索方式
- **并行检索**：同一查询同时发给向量搜索引擎和 BM25 搜索引擎，再将结果融合。
- **融合策略**：
  - 直接使用 RRF 合并排名。
  - 分别归一化后再加权求和。
- **适用场景**：
  - 纯语义检索：用户意图模糊、跨语言、同义词较多。
  - 关键词检索不可或缺：专业术语、产品型号、代码片段等必须精确匹配的场景。

## Embedding 模型的选择与调优

### 常用句子/段落级模型
- **OpenAI text-embedding-3-small/large**：通用性强，多语言，但有费用。
- **BGE 系列**（如 `bge-large-zh-v1.5`）：中文表现优秀，开源可微调。
- **M3E**：专为中文优化的 Embedding 模型，适合国内业务。
- **Jina Embeddings v2**：支持 8K 长文本，适合大块文档。

### 模型评估
- 使用 **MTEB**（Massive Text Embedding Benchmark）基准评估语义相似度、检索、分类等任务。
- 在自己的数据集上构建评估集，计算召回率、MRR（Mean Reciprocal Rank）等。

### 微调策略
- **对比学习微调**：使用（正例，负例）对，如 `sentence-transformers` 框架。
- **领域内数据增强**：利用 GPT 生成相似查询或借助用户点击日志构造正负例。
- **微调注意事项**：原模型能力可能退化，需混合通用数据和领域数据。

### 维度与存储成本
- 高维向量（如 1536 维）检索更慢、存储更大；低维（如 256~512 维）可借助降维技术（如 PCA、模型自带 Matryoshka 表示）减少开销。
- OpenAI `text-embedding-3` 支持 `dimensions` 参数，可直接输出指定维度。

## 生成阶段的高级优化模式

| 模式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Map‑Reduce** | 每份文档独立生成中间答案，再汇总合并 | 并行处理、适合大量文档 | Token 消耗大，汇总可能丢失信息 |
| **Refine** | 逐文档迭代精炼答案 | 上下文逐步累积，信息损失小 | 串行执行，速度慢 |
| **Map‑Rerank** | 每份文档打分，选择最高分文档的回答 | 简单直接 | 可能丢弃其他文档有用信息 |
| **Tree Summarize** | 递归合并摘要至单份 | 对超长文档处理友好 | 摘要可能遗漏细节 |

**选用建议**：
- 文档较少（1~3 个）：直接用一次生成。
- 文档多且无顺序依赖：Map‑Reduce。
- 需要保留详细逻辑：Refine。
- 超长单文档分块：Tree Summarize。

### 多轮问题的独立化改写 Prompt

### 复杂问题的子问题分解（Self‑Ask）
- **步骤**：LLM 自动将复杂问题拆解为若干子问题，逐一回答（可递归检索），最后整合。
- **优点**：提高多跳推理问题的准确率。
- **实现**：设计 Few‑shot 示例引导模型。

### HyDE（假设文档嵌入）
- **原理**：先让 LLM 生成一个假定的答案段落，再将该假答案进行 Embedding 检索真实文档。
- **适用**：当用户提问与文档内容在表述上差距较大时，可大幅提高召回率。
- **流程**：`用户问题 → LLM生成假设答案 → 嵌入假设答案 → 检索 → 生成最终回答`

## RAG 系统评估

### RAGAS 核心指标
- **Faithfulness（忠实度）**：答案中的陈述是否都能在检索到的上下文中找到依据。计算为 `（可验证陈述数 / 总陈述数）`。
- **Answer Relevancy（回答相关性）**：答案与问题的相关程度，通过构造反向问题计算相似度。
- **Context Recall（上下文召回率）**：真值答案中所需的信息能被检索到的比例。
- **Context Precision（上下文精确率）**：检索出的上下文中真正相关的比例。

### 自动评估与人工评估
- **自动评估**：用另一 LLM 评价，成本低、速度快，但可能有偏见，适合初期快速迭代。
- **人工评估**：对最终效果进行抽检，从准确性、流畅性、有用性等维度打分。
- **结合使用**：先自动初筛，再人工抽检疑难案例。

### 评估数据集构建
- 来源：真实用户日志、合成数据、众包标注。
- 每条数据包含：**问题**、**标准答案**、**相关参考文档块**。

## 工程化落地知识

### 增量索引
- **新增文档**：切分后生成 Embedding 并插入向量数据库，记录元数据（时间、来源）。
- **删除/更新**：通过元数据过滤找到旧向量并删除，再插入新向量。多数向量数据库支持 `delete` 操作。
- **定时同步**：监听知识库变更（如 Git 仓库、数据库），通过消息队列触发重新索引。

### 语义缓存
- **原理**：对查询的 Embedding 与缓存的嵌入做相似度匹配，超过阈值直接返回缓存答案，减少 LLM 调用。
- **工具**：GPTCache、Redis + `faiss` 等。
- **配置项**：相似度阈值、缓存过期时间、最大缓存条目。

### 权限与多租户隔离
- **元数据过滤**：在检索时为每个向量添加租户 ID、部门 ID 字段，查询时附加过滤条件。
- **分区/集合隔离**：为不同租户创建独立集合，物理隔离更彻底。

### 监控与调试
- **监控指标**：检索平均相似度、重排后相关率、端到端延迟、LLM 调用失败率。
- **调试手段**：记录每条查询的完整链路（检索结果、重排结果、最终提示词和回答），便于回溯分析。

### 成本优化
- **动态 chunk 大小**：根据文档复杂度选择不同 chunk 长度，避免过多冗余上下文。
- **动态路由 LLM**：简单问题用小模型（如 GPT‑3.5），复杂推理用大模型（如 GPT‑4）。
- **过滤与排序优先**：在召回阶段多花成本精准过滤，减少后续 LLM 调用次数。
