# 📌 第一部分：选择题速记（25分）

> 广大选择题**重复率极高**，以下都是真题高频考点，务必全部记住！

### 🎯 必背选择题30道

**1. 软件 = 程序 + 数据 + 文档**

**2. 软件危机的主观原因**：开发与维护的方法不正确

**3. 软件工程目的**：最终解决软件的**生产工程化**问题

**4. 瀑布模型**本质是**线性顺序**

**5. 维护**属于**运行期**，不属于开发期

**6. 计划期**任务 = 问题定义 + 可行性研究

**7. 可行性研究**要做**简化的、压缩的**需求分析

**8. 经济可行性**研究范围包括**效益分析**

**9. 可行性分析**必须给出**行或不行**的回答

**10. 软件维护副作用** = **因修改软件而造成的错误**

**11. 可维护性相互矛盾的特性** = **效率和可修改性**

**12. 需求分析四方面** = 需求获取 + 需求分析 + 编写SRS + **需求评审**

**13. 结构化分析工具** = **数据流图 + 数据字典**

**14. 数据流图合法图符**：数据流、加工、数据存储、外部实体。**没有控制流！**

**15. 结构化生命周期法基本假定**：需求能做到**早期冻结**

**16. 软件功能分解**属于**总体设计**阶段

**17. 概要设计输出** = **模块说明书**

**18. 一组模块都访问同一数据结构** = **公共耦合**

**19. 按特定次序执行元素** = **过程内聚**（不是顺序内聚！）

**20. 最强内聚** = **功能内聚**

**21. 耦合最弱（最好）** = **数据耦合**

**22. 耦合最强（最差）** = **内容耦合**

**23. 局部化** = 让关系密切的软件元素在物理上靠近

**24. 软件测试目的** = **发现软件的错误**（不是证明正确！）

**25. 黑盒测试**从**程序功能**观点出发，白盒测试从**程序内部逻辑**出发

**26. 回归测试** = 重新执行已做过的测试

**27. 集成测试**最好由**不属于该开发组的软件设计人员**承担

**28. 多态** = 不同对象收到相同消息产生不同动作

**29. 汽车和发动机** = **整体-部分**关系

**30. 扇入**反映程序重用率（扇入=有多少上级模块调用该模块）

### 🎯 补充高频考点

- 软件结构图形态特征反映重用率的是**扇入**
- 判定树和判定表用于描述**数据加工**
- 数据流图中数据流经文件时**数据流不必命名，有文件名就够了**
- 分层数据流图顶层描绘系统的**抽象**
- 程序三种基本控制结构：**顺序、选择、循环**，共同特点：**单入口单出口**
- 详细设计工具：**程序流程图、PAD、方框图、伪码**
- 功能设计属于**总体设计**（概要设计），不属于详细设计

# 📌 第二部分：名词解释必背（25分）

> **答题要求**：至少写3-4句话，5分的题要写到100字以上！

### 🔑 必背8个名词解释

**1. 软件工程（5分）**
将系统的、规范的、可量化的方法应用于软件的开发、运行和维护的过程；及上述方法的研究。软件工程包括三个要素：方法、工具和过程。

**2. 软件危机**
在计算机软件的开发和维护过程中所遇到的一系列严重问题。主要表现为：成本和进度估计不准确、用户不满意、质量靠不住、不可维护、缺少文档、成本比例上升、生产率提高缓慢。产生原因包括软件本身规模大、结构复杂，以及开发方法不正确。

**3. UML（统一建模语言）**
UML是用来对软件密集系统进行可视化建模的一种语言，是对面向对象开发系统的产品进行说明、可视化和文档编制的一种标准语言。其特点包括：①统一标准（融合Booch、OMT、OOSE）；②面向对象；③表达能力强大且可视化。

**4. 用例（Use Case）**
⑴ 从外部用户视角看，一个用例是执行者（actor）与目标软件系统之间一次典型的交互作用，其效果就是执行者在软件系统的帮助下完成了某项业务功能或达成了某项业务目标。
⑵ 从软件系统内部视角出发，一个用例代表着系统执行的一系列动作，动作执行的结果能够被外部执行者所察觉。

**5. 黑盒测试**
黑盒测试就是根据被测程序功能来进行测试，也称为功能测试或数据驱动测试。主要着眼于程序外部结构，不考虑内部逻辑结构，针对软件界面和软件功能进行测试。在已知产品应具有功能的情况下，通过测试检验每个功能是否都能正常使用。常用技术包括等价类划分法、边界值分析法、错误推测法和因果图法。

**6. 白盒测试**
白盒测试以程序的结构为依据，又称为结构测试。按照程序的内部结构测试程序，检验程序中的每条通路是否都能按预定要求正确工作。要求测试人员对程序的逻辑结构有清楚了解，甚至掌握源程序的所有细节。主要方法包括逻辑覆盖测试和路径测试。

**7. 软件体系结构（5分）**
软件体系结构包括三大要素：组件、连接件和约束。连接件表示组件之间的连接和交互关系；约束表示组件中的元素应满足的条件，以及组件经由连接件组装成更大模块时应满足的条件。

**8. 软件质量**
软件产品满足规定的和隐含的与需求能力有关的全部特征和特性。包括六大特性：功能性、可靠性、易用性、效率、可维护性、可移植性。

### 🔑 补充名词解释

**9. 完善性维护**
根据用户在软件使用过程中提出的一些新需求实施的维护活动。在所有维护类型中占比最大（约50%）。

**10. 数据字典**
数据字典是关于数据的信息的集合，也就是对数据流图中包含的所有元素的定义的集合。数据字典的作用是在软件分析和设计过程中给人提供关于数据的描述信息，它与数据流图共同构成系统的逻辑模型。

**11. 可行性分析**
可行性分析是在系统开发的早期，对项目在技术上、经济上、操作上是否可行进行的论证工作。它是决定该系统是否开发的决策依据，必须给出"行或不行"的回答。

# 📌 第三部分：简答题必背模板（30分）

### 📝 模板1：生命周期模型对比（必考！约12分）

| 模型       | 优点                  | 缺点                  | 适用条件       |
| -------- | ------------------- | ------------------- | ---------- |
| **瀑布模型** | 顺序性和阶段性，文档驱动，易于管理   | 缺乏灵活性，逆转性差，不适应需求变化  | 需求明确、稳定的项目 |
| **快速原型** | 用户早期参与，减少需求风险，开发速度快 | 原型可能被当作最终产品，不利于创新   | 需求不明确的项目   |
| **增量模型** | 用户可早期使用部分功能，分阶段提交   | 需要开放的体系结构，集成风险      | 可分模块的中大型项目 |
| **螺旋模型** | 强调风险分析，适合大型项目       | 风险评估成本高，需丰富经验       | 大型、高风险项目   |
| **喷泉模型** | 面向对象，迭代无间隙，提高开发效率   | 文档管理困难，需严格管理        | 面向对象开发     |
| **敏捷模型** | 响应变化快，客户满意度高，开发效率高  | 对团队要求高，文档不足，不适合大型项目 | 小团队，需求多变   |

### 📝 模板2：需求类型（5分）

⑴ **功能需求**（2分）：指利益相关方要求目标软件系统应该具有的功能，还包括在业务处理过程中完成这些功能时必须遵守的约定或限制。
⑵ **质量需求**（1分）：利益相关方对目标软件系统的质量要求。
⑶ **约束性需求**（2分）：利益相关方对项目预算、完成时间、技术选型、遵循的标准与规范等方面的要求，以及由预期的开发、运行环境特征导致的约束。

### 📝 模板3：内聚7级（8分）

内聚度表示一个模块内部各成分彼此关联的紧密程度。表现形式从低到高有7种：
⑴ **偶然性内聚**：模块内各成分联系很松散
⑵ **逻辑性内聚**：完成多项功能，逻辑上有某种相关性
⑶ **时间性内聚**：诸任务必须在同一时间段内执行
⑷ **过程性内聚**：各功能必须按特定次序执行
⑸ **通信性内聚**：各成份对数据结构同一区域进行操作
⑹ **顺序性内聚**：各处理与同一功能密切相关，必须依序执行
⑺ **功能性内聚**：各成份协同完成单一功能（最强）

### 📝 模板4：软件危机表现（5分）

**表现形式**（7条）：
①成本和进度估计不准确 ②用户不满意 ③质量靠不住 ④不可维护 ⑤缺少文档 ⑥成本比例上升 ⑦生产率提高缓慢

**产生原因**：
①软件本身规模大、结构复杂 ②开发管理困难 ③开发费用增加 ④开发技术落后 ⑤生产方式落后 ⑥开发工具落后

**缓解途径**：
①正确认识软件 ②推广成功技术和方法 ③使用软件工具和开发环境 ④加强软件管理

### 📝 模板5：增量式开发优缺点（10分）

**优点：**
⑴ 按增量持续发布新版本，及时获得客户反馈，调整后续开发策略
⑵ 软件需求确定，可先设计体系结构，增量开发保持良好架构

**缺点：**
⑴ 增量规模不能大（不超过20k行代码），否则暴露瀑布模型缺点
⑵ 将需求分解成增量序列需对系统需求十分了解，有顶层设计经验
⑶ 多数系统需基本服务，如何定义和实现增量比较困难

### 📝 模板6：用例模型为何还需分析模型（6分）

⑴ 分析模型比用例模型更结构化、更清晰直观，构建过程也是深入理解用例的过程，同时也是剔除自然语言描述中模糊性和不一致性的过程。
⑵ 分析模型是用例模型与设计模型之间的"桥梁"，比用例模型更接近设计模型，更适合软件设计师设计系统结构、构思求解算法，更易于为不太熟悉业务的设计师所理解。

# 📌 第四部分：白盒测试计算题攻略（10分）

## 五种覆盖方法（从弱到强）

```
语句覆盖 → 判定覆盖 → 条件覆盖 → 判定/条件覆盖 → 条件组合覆盖
  最弱                                                      最强
```

| 覆盖方法 | 定义 |
|----------|------|
| **语句覆盖** | 每条可执行语句至少执行一次 |
| **判定覆盖** | 每个判定的真假分支至少执行一次 |
| **条件覆盖** | 每个判定中每个条件的真假至少出现一次 |
| **判定/条件覆盖** | 同时满足判定覆盖和条件覆盖 |
| **条件组合覆盖** | 每个判定中条件的各种组合至少出现一次 |

## 经典例题（必须会做！）

**程序：**
```
IF (A > 1) AND (B = 0) THEN X = X / A
IF (A = 2) OR  (X > 1) THEN X = X + 1
```

**4个条件**：①A>1 ②B=0 ③A=2 ④X>1

### (1) 语句覆盖
只需让两行代码都执行到。

| 用例 | A | B | X | 路径 |
|------|---|---|---|------|
| 1 | 2 | 0 | 4 | 两行都执行 ✓ |

**最少用例数：1个**

### (2) 判定覆盖
每个判定的T和F至少各出现一次。

| 用例 | A | B | X | 判定1 | 判定2 |
|------|---|---|---|-------|-------|
| 1 | 2 | 0 | 4 | T | T |
| 2 | 1 | 1 | 1 | F | F |

**最少用例数：2个**

### (3) 条件覆盖
4个条件的T和F至少各一次。

| 用例 | A | B | X | A>1 | B=0 | A=2 | X>1 |
|------|---|---|---|-----|-----|-----|-----|
| 1 | 2 | 0 | 4 | T | T | T | T |
| 2 | 1 | 1 | 1 | F | F | F | F |

**注意：条件覆盖不一定满足判定覆盖！**

### (4) 判定/条件覆盖
同时满足判定覆盖和条件覆盖 → 同上表

### (5) 条件组合覆盖
判定1的4种组合 + 判定2的4种组合

| 用例 | A | B | X | 判定1组合 | 判定2组合 |
|------|---|---|---|----------|----------|
| 1 | 2 | 0 | 2 | (T,T) | (T,T) |
| 2 | 2 | 1 | 1 | (T,F) | (T,F) |
| 3 | 1 | 0 | 4 | (F,T) | (F,T) |
| 4 | 1 | 1 | 1 | (F,F) | (F,F) |

## 环形复杂度

**公式**：V(G) = E - N + 2（E=边数，N=节点数）
**或**：V(G) = 判定节点数 + 1
**意义**：V(G) = 独立路径条数 = 至少需要的测试用例数

# 📌 第五部分：设计题作画指南（35分）

## 题型1：用例图（约10分）

**作画步骤**：
1. **识别参与者**（Actor）：与系统交互的外部实体（人形图标）
2. **识别用例**（Use Case）：系统提供的服务（椭圆）
3. **画关联**：参与者与用例之间用实线连接
4. **标注关系**：
   - `<<include>>`：A执行时**必须**执行B（虚线箭头从A到B）
   - `<<extend>>`：在特定条件下B**扩展**A（虚线箭头从B到A）

**经典场景**：图书管理系统
- 参与者：读者、图书管理员
- 用例：借书、还书、查询、续借、预约、入库、出库
- 关系：借书 --`<<include>>`--> 验证身份；还书 --`<<extend>>`--> 缴纳罚款

## 题型2：活动图（约10分）

**符号**：
- 活动 → 圆角矩形
- 判断 → 菱形（标注条件，出口标注[是]/[否]）
- 分叉/汇合 → 粗横线
- 开始 → 实心圆 ●
- 结束 → 牛眼 ◎
- 泳道 → 竖向分区（不同参与者）

**经典场景**：学生选课
```
[开始] → [登录系统] → [浏览课程] → [选择课程]
    → <课程已满?>
        → 是 → [提示已满] → [重新选择]
        → 否 → <已选该课?>
            → 是 → [提示已选]
            → 否 → [提交选课] → [系统确认] → [结束]
```

## 题型3：ER图（约15分）

**三要素**：
- 实体 → **矩形** □
- 属性 → **椭圆** ○
- 关系 → **菱形** ◇

**关系基数**：
- 1:1（一对一）、1:N（一对多）、M:N（多对多）

**经典场景**：学生选课系统
- 实体：学生（学号、姓名、性别）、课程（课程号、课程名、学分）
- 关系：选课（M:N），属性：成绩
- 其他关系：班级 1:N 学生，院系 1:N 专业，专业 1:N 班级

## 题型4：顺序图（如考到）

- 横轴：对象（参与者和系统对象）
- 纵轴：时间（从上到下）
- 消息：带箭头的实线
- 生命线：虚线
- 激活条：窄矩形

## 题型5：状态图（如考到）

- 状态 → 圆角矩形
- 初始状态 → 实心圆 ●
- 终止状态 → 牛眼 ◎
- 转换 → 带箭头线，标注事件/条件

# 📌 第七部分：易混淆概念速查表

| 易混点 | 正确答案 |
|--------|---------|
| 过程内聚 vs 顺序内聚 | 过程=按特定次序；顺序=一个输出是下一个输入 |
| 判定覆盖 vs 条件覆盖 | 互相独立！判定覆盖不一定包含条件覆盖 |
| include vs extend | include=必须执行；extend=条件扩展 |
| 驱动模块 vs 桩模块 | 驱动=模拟调用者（上级）；桩=模拟被调用者（下级） |
| Alpha vs Beta测试 | Alpha=开发方场所；Beta=用户场所 |
| 黑盒 vs 白盒 | 黑盒=功能；白盒=结构 |
| 内容耦合 vs 数据耦合 | 内容=最差（直接访问内部）；数据=最好（只传参数） |
| 功能内聚 vs 偶然内聚 | 功能=最强；偶然=最弱 |
| 完善性维护 vs 改正性维护 | 完善=增加新功能（占比最大）；改正=修复Bug |
| 计划期 vs 开发期 vs 运行期 | 计划=问题定义+可行性；开发=需求+设计+编码+测试；运行=维护 |
