# 第一章 软件工程概述 ⭐⭐⭐⭐⭐

## 1.1 软件的定义与特点 ⭐⭐⭐⭐

**软件** = **程序** + **数据** + **文档**

**软件的5个特点（必背！）：**

| 序号 | 特点 | 说明 |
|------|------|------|
| 1 | **逻辑性** | 软件是逻辑产品，不是物理产品，看不到摸不着 |
| 2 | **不会磨损** | 软件不会像硬件那样因使用而老化磨损 |
| 3 | **开发方式** | 主要是开发而非制造，大部分成本在开发阶段 |
| 4 | **复杂性** | 软件系统规模庞大、逻辑复杂 |
| 5 | **可复用性** | 软件可以被复制和重用，复制成本几乎为零 |

## 1.2 软件危机 ⭐⭐⭐⭐⭐

### 定义
**软件危机**：在计算机软件的**开发**和**维护**过程中所遇到的一系列严重问题。

### 7个表现（必背！）

| 序号 | 表现 |
|------|------|
| 1 | 开发成本和进度**估计不准确** |
| 2 | 用户对"已完成的"软件系统**不满意** |
| 3 | 软件产品的**质量**靠不住 |
| 4 | 软件**不可维护** |
| 5 | 软件**没有适当的文档资料** |
| 6 | 软件成本在计算机系统总成本中所占比例**逐年上升** |
| 7 | 软件开发生产率提高的速度跟不上计算机应用迅速普及深入的趋势 |

### 原因
- **客观原因**：软件本身特点（逻辑产品、规模庞大、复杂）
- **主观原因**：开发与维护的方法不正确

### 缓解途径
1. 对计算机软件有**正确认识**
2. 推广使用成功的**开发技术和方法**
3. 使用**软件工具**和**开发环境**
4. 加强**软件管理**

## 1.5 七种过程模型对比表 ⭐⭐⭐⭐⭐（必考简答题！）

| 模型 | 原理 | 优点 | 缺点 | 适用条件 |
|------|------|------|------|----------|
| **瀑布模型** | 线性顺序，阶段间有反馈 | 文档驱动，易于管理 | 风险延迟，不适应需求变化 | 需求明确、稳定的项目 |
| **快速原型** | 快速构建原型→用户评估→迭代改进 | 用户早期参与，减少需求风险 | 原型可能被当作最终产品 | 需求不明确的项目 |
| **增量模型** | 分批交付，每次增加功能 | 用户可早期使用部分功能 | 需要开放的体系结构 | 可分模块的中大型项目 |
| **螺旋模型** | 迭代+风险分析 | 强调风险分析，适合大型项目 | 风险评估成本高 | 大型、高风险项目 |
| **喷泉模型** | 面向对象，迭代无间隙 | 支持迭代开发和复用 | 文档管理困难 | 面向对象开发 |
| **敏捷模型** | 小步快跑，拥抱变化 | 响应变化快，客户满意度高 | 对团队要求高，文档不足 | 小团队，需求多变 |
| **微软模型** | 改进的瀑布+里程碑 | 实用性强，结合瀑布和迭代 | 针对特定开发环境 | 中大型商业软件 |

## 1.7 敏捷方法 ⭐⭐⭐⭐

3. **客户合作** 高于 合同谈判

6. 可工作软件是进度的首要度量标准
7. 可持续开发，保持稳定节奏
8. 关注技术卓越和良好设计
9. 简洁——减少不必要工作
10. 最好的架构、需求和设计来自自组织团队
11. 团队定期反思如何更高效
12. 拥抱变化

# 第二章 面向对象与UML ⭐⭐⭐⭐⭐

## 2.1 面向对象基本概念 ⭐⭐⭐⭐⭐

| 概念 | 定义 | 举例 |
|------|------|------|
| **对象** | 系统中用来描述客观事物的一个实体，由**属性（数据）**和**方法（操作）**组成 | 学生对象：属性=姓名/学号/成绩，方法=选课/查成绩 |
| **类** | 具有相同属性和方法的**一组对象的集合** | "学生"类包含所有学生对象 |
| **封装** | 将对象的属性和方法结合在一起，隐藏内部细节，只暴露接口 | 手机：用户只看到界面，看不到内部电路 |
| **继承** | 子类自动拥有父类的属性和方法 | "大学生"继承"学生"类的所有属性 |
| **多态** | 同一操作在不同对象上有不同表现 | "画"操作：圆→画圆，矩形→画矩形 |
| **消息** | 对象之间进行通信的方式 | 用户调用"学生.查成绩()"就是发送消息 |

**面向对象 = 对象 + 类 + 继承 + 消息通信**

## 2.2 UML ⭐⭐⭐⭐⭐

### UML定义
**统一建模语言（Unified Modeling Language）**，是一种用于对软件密集型系统的产品进行**可视化、规约、构造和文档化**的标准建模语言。

### UML 3个特点
1. **统一的标准**：融合了Booch、OMT、OOSE方法
2. **面向对象**：支持面向对象开发
3. **可视化、表达能力强**

### UML九种图 ⭐⭐⭐⭐⭐

| 分类 | 图名 | 定义 |
|------|------|------|
| **用例图** | 用例图 | 描述系统功能和参与者之间的关系 |
| **静态图** | 类图 | 描述类与类之间的静态关系 |
| | 对象图 | 描述某一时刻类的具体实例及实例间的关系 |
| **行为图** | 状态图 | 描述对象在其生命周期内所经历的状态序列 |
| | 活动图 | 描述活动的执行顺序（类似流程图） |
| **交互图** | 顺序图（序列图） | 按时间顺序描述对象间的交互 |
| | 协作图 | 描述对象间的协作关系和消息连接 |
| **实现图** | 构件图 | 描述软件构件及其依赖关系 |

## 2.3 用例之间的关系 ⭐⭐⭐⭐⭐（必考设计题）

### 包含关系（include）
**A <<include>> B**：执行A**必须**执行B

```
┌─────────┐         <<include>>        ┌─────────┐
│ 用例A    │ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ →  │ 用例B    │
└─────────┘                            └─────────┘
  例：登录 ──<<include>>──→ 验证身份
```

### 扩展关系（extend）
**B <<extend>> A**：在特定条件下，A可以被B扩展

```
┌─────────┐         <<extend>>        ┌─────────┐
│ 用例B    │ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ → │ 用例A    │
└─────────┘                            └─────────┘
  例：显示错误 ──<<extend>>──→ 登录
```

### 继承关系
子用例继承父用例的行为，可重写或扩展。

```
        ┌─────────┐
        │ 父用例   │
        └────┬────┘
             │
       ┌─────┴─────┐
       ▼           ▼
  ┌─────────┐ ┌─────────┐
  │ 子用例A  │ │ 子用例B  │
  └─────────┘ └─────────┘
```

# 第四章 软件需求分析 ⭐⭐⭐⭐⭐

## 4.1 需求分析的目的与重要性 ⭐⭐⭐

- **目的**：确定系统必须完成哪些工作，也就是对目标系统提出**完整、准确、清晰、具体**的要求
- **重要性**：需求分析是软件开发的**基础**，需求错误是最昂贵的错误

## 4.2 需求类型 ⭐⭐⭐⭐⭐

| 类型 | 定义 | 举例 |
|------|------|------|
| **功能需求** | 系统必须完成的功能 | 用户能登录、查询成绩 |
| **质量需求（非功能需求）** | 系统的性能、可靠性等质量属性 | 响应时间<3秒，可用率99.9% |
| **约束性需求** | 系统开发和运行的约束条件 | 必须运行在Linux上，6个月内完成 |

## 4.3 需求分析的过程 ⭐⭐⭐

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 需求获取  │→│ 需求分析  │→│ 需求文档  │→│ 需求评审  │
│ (获取)   │  │ (分析)   │  │ (编写)   │  │ (验证)   │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

## 4.4 需求分析方法对比 ⭐⭐⭐⭐

| 方法 | 核心思想 | 工具 | 适用场景 |
|------|----------|------|----------|
| **结构化分析SA** | 分解、抽象，自顶向下 | DFD、DD、ER图 | 数据处理系统 |
| **面向对象分析OOA** | 对象、类、继承、消息 | UML用例图、类图 | 面向对象系统 |
| **快速原型法** | 快速构建原型获取需求 | 原型工具 | 需求不明确时 |

## 4.6 数据流图DFD ⭐⭐⭐⭐

# 第五章 软件设计 ⭐⭐⭐⭐⭐

## 5.1 软件设计的3条准则 ⭐⭐⭐

1. **分而治之**：将大问题分解为小问题
2. **提高抽象层次**：隐藏细节
3. **提高可复用性**：设计可复用的模块

## 5.2 软件设计原理 ⭐⭐⭐⭐

| 原理 | 说明 |
|------|------|
| **抽象** | 忽略与当前目标无关的方面，集中关注当前目标 |
| **逐步求精** | 将宏观问题逐步分解细化 |
| **模块化** | 将系统分解为独立的模块 |
| **信息隐藏** | 模块内部的实现细节对外不可见 |

## 5.3 耦合（从高到低5级）⭐⭐⭐⭐⭐（必考！）

> **耦合**：模块之间相互依赖的程度，越低越好

```
高 ──────────────────────────────────────── 低
内容  公共  外部  控制  数据   ←  耦合强度
```

| 级别 | 耦合类型 | 定义 | 举例 |
|------|----------|------|------|
| 最高 | **内容耦合** | 一个模块直接访问另一个模块的内部数据或代码 | goto跳入另一模块 |
| ↓ | **公共耦合** | 多个模块共用全局数据 | 都访问同一个全局变量 |
| ↓ | **外部耦合** | 多个模块访问同一外部数据格式/协议 | 都用同一文件格式 |
| ↓ | **控制耦合** | 一个模块传"控制标志"给另一模块，影响其内部逻辑 | 传flag参数控制另一模块执行路径 |
| 最低 | **数据耦合** | 模块间只通过参数传递数据 | 函数调用传递参数 |

**设计原则：尽量使用数据耦合，少用控制耦合，限制公共耦合，完全不用内容耦合。**

## 5.4 内聚（从低到高7级）⭐⭐⭐⭐⭐（必考！）

> **内聚**：模块内部各元素之间的关联程度，越高越好

```
低 ──────────────────────────────────────── 高
偶然  逻辑  时间  过程  通信  顺序  功能   ←  内聚程度
```

| 级别 | 内聚类型 | 定义 | 举例 |
|------|----------|------|------|
| 最低 | **偶然内聚** | 模块内各处理无关联，纯属巧合放在一起 | 把几条无关联语句放到一个模块 |
| ↓ | **逻辑内聚** | 逻辑相似的功能放在同一模块 | 打印报表和显示屏幕放一起 |
| ↓ | **时间内聚** | 同一时间内完成的任务放在同一模块 | 系统初始化模块 |
| ↓ | **过程内聚** | 按特定次序执行的处理放在同一模块 | 先读数据再计算 |
| ↓ | **通信内聚** | 使用相同数据的处理放在同一模块 | 都对同一数据库操作 |
| ↓ | **顺序内聚** | 一个处理的输出是另一个处理的输入 | 先计算再格式化再输出 |
| 最高 | **功能内聚** | 模块只完成一个功能 | 计算平均分模块 |

## 5.5 耦合内聚设计原则 ⭐⭐⭐⭐⭐

# 第七章 软件测试 ⭐⭐⭐⭐⭐（最重要！占分最大！）

## 7.1 软件测试定义与目的 ⭐⭐⭐⭐⭐

### Myers的3个观点（必背！）
1. 测试是程序的**执行过程**，目的在于发现错误
2. 好的测试用例能**发现尚未发现的错误**
3. 成功的测试是**发现了至今未发现的错误**的测试

## 7.2 测试原则8条 ⭐⭐⭐⭐

1. 测试用例应由**测试输入数据**和**预期输出结果**两部分组成
2. 程序员应避免**检查自己**的程序
3. 编写测试用例时应包含**合理的**和**不合理的**输入条件
4. 充分注意测试中的**群集现象**（80%的错误集中在20%的模块中）
5. 严格执行测试计划，**排除测试的随意性**
6. 对每一个测试结果做**全面检查**
7. 妥善保存测试计划、用例、出错统计和最终分析报告
8. **不可能完全测试**（穷举测试不可能）

## 7.3 测试的4个性质 ⭐⭐⭐

| 性质 | 说明 |
|------|------|
| **挑剔性** | 测试是为了发现错误，要"挑剔" |
| **复杂性** | 测试工作量大、难度高 |
| **不彻底性** | 不可能完全测试所有情况 |
| **经济性** | 要在有限资源下达到最大测试效果 |

## 7.4 V模型 ⭐⭐⭐⭐⭐（必背！）

```
V模型（开发阶段与测试阶段的对应关系）

  开发阶段                                    测试阶段
                                              ┌──────────────┐
                                              │  验收测试     │ ←── 用户参与
  ┌──────────────┐                            ├──────────────┤
  │  需求分析     │                            │  系统测试     │ ←── 验证需求
  │(定义"做什么") │ ←─────────────────────────│(整个系统测试) │
  ├──────────────┤                            ├──────────────┤
  │  概要设计     │                            │  集成测试     │ ←── 验证接口
  │(定义"怎么做") │ ←─────────────────────────│(模块间接口)   │
  ├──────────────┤                            ├──────────────┤
  │  详细设计     │                            │  单元测试     │ ←── 验证逻辑
  │(具体算法)    │ ←─────────────────────────│(单个模块/函数)│
  ├──────────────┤                            └──────────────┘
  │    编码      │
  │(写代码)      │
  └──────────────┘

对应关系：
  需求分析 ←→ 系统测试：验证系统是否满足需求
  概要设计 ←→ 集成测试：验证模块间接口是否正确
  详细设计 ←→ 单元测试：验证单个模块逻辑是否正确
```

**对应关系：**
- 需求分析 ←→ 系统测试（验证是否满足需求）
- 概要设计 ←→ 集成测试（验证模块间接口）
- 详细设计 ←→ 单元测试（验证单个模块）

### W/H/X/前置模型简述

| 模型 | 特点 |
|------|------|
| **W模型** | 测试与开发同步，双V，测试贯穿整个生命周期 |
| **H模型** | 测试是独立的流程，可随时插入 |
| **X模型** | 探索性测试，强调缺陷修复后的回归测试 |
| **前置模型** | 将测试活动提前到开发早期 |

## 7.5 静态测试 vs 动态测试 ⭐⭐⭐

| 类型 | 说明 | 方法 |
|------|------|------|
| **静态测试** | 不运行程序，检查代码/文档 | 代码审查、走查、静态分析 |
| **动态测试** | 运行程序，检查输出 | 黑盒测试、白盒测试 |

## 7.6 黑盒测试 ⭐⭐⭐⭐⭐（必考选择题）

**定义**：把程序看作一个黑盒子，不考虑内部逻辑，只根据**输入/输出**来测试。

### 方法1：等价分类法 ⭐⭐⭐⭐⭐

**原理**：将输入数据划分为若干等价类，从每个类中选一个代表。

**PPT例7.1 招工年龄问题：**
> 招工条件：18～35岁之间的初中以上文化程度的青年

**分析输入条件：**
- 年龄：18~35岁
- 文化程度：初中及以上

**划分等价类：**

| 输入条件 | 有效等价类 | 无效等价类 |
|----------|-----------|-----------|
| 年龄 | ① 18~35（如25） | ② <18（如10）③ >35（如50） |
| 文化程度 | ④ 初中及以上（如高中） | ⑤ 小学 |

**设计测试用例：**
| 用例 | 年龄 | 文化程度 | 期望结果 | 覆盖等价类 |
|------|------|----------|----------|-----------|
| 1 | 25 | 高中 | 录取 | ①④ |
| 2 | 10 | 高中 | 不录取 | ②④ |
| 3 | 50 | 高中 | 不录取 | ③④ |
| 4 | 25 | 小学 | 不录取 | ①⑤ |

### 方法2：边界值分析法 ⭐⭐⭐⭐⭐

**原理**：选取**边界值**及其**附近值**作为测试用例。

**PPT例7.2：**
> 输入值范围为 [1, 100]

测试用例：0, 1, 2, 99, 100, 101（取边界和边界两侧）

**边界值分析法要点：**
- 如果输入范围为 [a, b]，测试 a-1, a, a+1, b-1, b, b+1
- 如果输入个数为 [m, n]，测试 m-1, m, m+1, n-1, n, n+1

### 方法3：错误猜测法
基于经验和直觉推测程序中可能存在的错误。

### 方法4：因果图法
将输入条件（因）和输出结果（果）用因果图表示，再转换为判定表。

## 7.7 白盒测试 ⭐⭐⭐⭐⭐（必考计算题10分！）

**定义**：把程序看作一个透明的盒子，根据程序的**内部逻辑结构**来测试。

### 5种覆盖方法（从弱到强）⭐⭐⭐⭐⭐

```
弱 ───────────────────────────────────────────→ 强
语句覆盖 → 判定覆盖 → 条件覆盖 → 判定/条件覆盖 → 条件组合覆盖
```

| 覆盖方法 | 含义 | 强度 |
|----------|------|------|
| **语句覆盖** | 每条可执行语句至少执行一次 | 最弱 |
| **判定覆盖** | 每个判定的真假分支至少执行一次 | 较弱 |
| **条件覆盖** | 每个判定中的每个条件的真假至少出现一次 | 中等 |
| **判定/条件覆盖** | 同时满足判定覆盖和条件覆盖 | 较强 |
| **条件组合覆盖** | 每个判定中条件的各种组合至少出现一次 | 最强 |

### 经典PPT例题 ⭐⭐⭐⭐⭐（必须会做！）

**程序段：**
```
IF (A > 1) AND (B = 0) THEN
    X = X / A
IF (A = 2) OR (X > 1) THEN
    X = X + 1
```

**流程图：**
```
          ┌─────────┐
          │ 入口     │
          └────┬────┘
               ▼
       ┌──────────────┐
      ╱ A>1 AND B=0 ? ╲──否──┐
     ╱                    ╲    │
    ▼ 是                    │   │
┌──────────┐               │   │
│ X = X/A  │               │   │
└────┬─────┘               │   │
     ▼                      │   │
     ├──────────────────────┘   │
     ▼                          │
    ┌──────────────┐            │
   ╱ A=2 OR X>1 ?  ╲──否──┐   │
  ╱                    ╲    │   │
 ▼ 是                    │   │   │
┌──────────┐            │   │   │
│ X = X+1  │            │   │   │
└────┬─────┘            │   │   │
     ▼                   │   │   │
     ├───────────────────┘   │   │
     ▼                       │   │
┌──────────┐                 │   │
│  出口     │←───────────────┘   │
└──────────┘←───────────────────┘
```

**4个判定条件：**
- 第1个IF：A>1 (T/F)，B=0 (T/F)
- 第2个IF：A=2 (T/F)，X>1 (T/F)

#### (1) 语句覆盖
只需让每条语句都执行到。

**测试用例：** A=2, B=0, X=4（任意初始值均可）
- 路径：A>1✓ B=0✓ → X=X/A=2 → A=2✓ → X=X+1=3
- 覆盖路径：a→c→e

**最少用例数：1个**

#### (2) 判定覆盖
每个判定的T和F至少各出现一次。

| 用例 | A | B | X | 判定1结果 | 判定2结果 | 覆盖路径 |
|------|---|---|---|----------|----------|---------|
| 1 | 2 | 0 | 4 | T | T | a→c→e |
| 2 | 1 | 1 | 1 | F | F | a→b→d→f |

**最少用例数：2个**

#### (3) 条件覆盖
每个条件的T和F至少各一次。

4个条件：①A>1 ②B=0 ③A=2 ④X>1

| 条件 | T | F |
|------|---|---|
| A>1 | A>1 (如A=2) | A≤1 (如A=1) |
| B=0 | B=0 | B≠0 (如B=1) |
| A=2 | A=2 | A≠2 (如A=3) |
| X>1 | X>1 (如X=2) | X≤1 (如X=1) |

| 用例 | A | B | X | A>1 | B=0 | A=2 | X>1 |
|------|---|---|---|-----|-----|-----|-----|
| 1 | 2 | 0 | 4 | T | T | T | T |
| 2 | 1 | 1 | 1 | F | F | F | F |

**注意：条件覆盖不一定满足判定覆盖！**

#### (4) 判定/条件覆盖
同时满足判定覆盖和条件覆盖。

| 用例 | A | B | X | 判定1 | 判定2 |
|------|---|---|---|-------|-------|
| 1 | 2 | 0 | 4 | T | T |
| 2 | 1 | 1 | 1 | F | F |

#### (5) 条件组合覆盖
每个判定中各条件的所有组合。

**判定1的组合：**
| 组合 | A>1 | B=0 | A,B取值 |
|------|-----|-----|---------|
| 1 | T | T | A=2, B=0 |
| 2 | T | F | A=2, B=1 |
| 3 | F | T | A=1, B=0 |
| 4 | F | F | A=1, B=1 |

**判定2的组合：**
| 组合 | A=2 | X>1 | 取值 |
|------|-----|-----|------|
| 1 | T | T | A=2, X=2 |
| 2 | T | F | A=2, X=1 |
| 3 | F | T | A=3, X=2 |
| 4 | F | F | A=3, X=1 |

**用例设计：**

| 用例 | A | B | X | 覆盖判定1组合 | 覆盖判定2组合 |
|------|---|---|---|--------------|--------------|
| 1 | 2 | 0 | 2 | 组合1(T,T) | 组合1(T,T) |
| 2 | 2 | 1 | 1 | 组合2(T,F) | 组合2(T,F) |
| 3 | 1 | 0 | 4 | 组合3(F,T) | 组合3(F,T) —注意此时X不变 |
| 4 | 1 | 1 | 1 | 组合4(F,F) | 组合4(F,F) |

### 路径测试与环形复杂度 ⭐⭐⭐⭐

**环形复杂度公式：**
> **V(G) = E - N + 2**
> - E = 边数（程序图中的弧线数）
> - N = 节点数（程序图中的节点数）

**也可以用：** V(G) = 判定节点数 + 1

**意义：** V(G) = 程序中**独立路径**的条数 = 至少需要的测试用例数

### 驱动模块和桩模块 ⭐⭐⭐⭐

| 模块 | 作用 | 模拟对象 |
|------|------|---------|
| **驱动模块** | 调用被测模块 | 模拟被测模块的**调用者**（上级） |
| **桩模块** | 被被测模块调用 | 模拟被测模块的**被调用者**（下级） |

```
驱动模块 ──调用──→ 被测模块 ──调用──→ 桩模块
(模拟上层)         (被测单元)         (模拟下层)
```

### 集成测试策略 ⭐⭐⭐⭐

| 策略 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **一次性集成** | 所有模块一次组装测试 | 简单快速 | 错误定位困难 |
| **自顶向下** | 从主模块开始逐层加入 | 不需要驱动模块，早期可见整体 | 需要大量桩模块 |
| **自底向上** | 从底层模块开始逐层组装 | 不需要桩模块 | 需要驱动模块，顶层最后测试 |
| **混合策略** | 上层自顶向下，下层自底向上 | 结合两者优点 | 实现较复杂 |

### Alpha测试 vs Beta测试 ⭐⭐⭐

| 类型 | 地点 | 人员 | 环境 |
|------|------|------|------|
| **Alpha测试** | 开发方场所 | 用户在开发者指导下测试 | 受控环境 |
| **Beta测试** | 用户场所 | 用户自己测试 | 实际使用环境 |

| **归纳法** | 从个别错误现象归纳出一般规律 |
| **演绎法** | 从一般原理出发推断出错误原因 |
| **回溯法** | 从错误点沿程序逻辑往回查找 |

# 第八章 软件维护 ⭐⭐⭐⭐

## 8.1 软件维护定义

在软件交付使用之后，为改正错误或满足新需求而修改软件的过程。

## 8.2 四种维护类型 ⭐⭐⭐⭐⭐

| 类型 | 目的 | 占比 | 举例 |
|------|------|------|------|
| **改正性维护** | 修复发现的错误 | ~20% | 修复Bug |
| **完善性维护** | 增强功能、提高性能 | ~50%（最大） | 增加新功能模块 |
| **适应性维护** | 适应环境变化 | ~25% | 移植到新操作系统 |
| **预防性维护** | 预防未来可能的问题 | ~5% | 重构代码 |

> **完善性维护占比最大！** 这是选择题常考点。

## 8.3 软件可维护性 ⭐⭐⭐

**定义**：软件能够被理解、校正、适应和增强的难易程度。

**提高方法：**
1. 建立明确的软件质量目标
2. 使用先进的软件开发技术
3. 进行明确的质量保证审查

## 二、名词解释精选5道（每题5分）

**1. 软件工程**
> 将**系统化、规范化、可度量**的方法应用于软件的**开发、运行和维护**，即将**工程**应用于软件。

**2. 用例（Use Case）**
> 描述系统执行的**一系列动作**，这些动作能产生对特定参与者有**价值的可观测结果**。

**3. 黑盒测试**
> 把程序看作一个**黑盒子**，完全不考虑程序的内部结构和处理过程，只依据程序的**规格说明**，检查程序的功能是否符合**功能规格说明**的要求。

**4. 白盒测试**
> 把程序看成一个装在**透明的白盒子**里，测试者**完全了解**程序的结构和处理过程，根据程序的**内部逻辑**来设计测试用例。

**5. 数据字典**
> 对数据流图中包含的所有元素的**定义和说明**的集合，是DFD的**配套工具**。

**补充：软件体系结构**
> 软件体系结构是软件系统的**组件**、组件之间的**连接件**以及组件与连接件之间的**约束**的总体描述。

**补充：可行性分析**
> 在系统开发前，从**技术、经济、操作**等方面分析项目是否可行，以确定项目是否值得开发。
