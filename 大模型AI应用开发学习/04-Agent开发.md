# Function Calling与MCP协议
## 一、Function Calling
### 要解决的问题
传统大模型智能聊天，没有工具调用功能：
1. **无法感知环境**：无法与外部数据源交互
2. **无法改变环境**：无法帮用户实际执行任务
### 如何解决问题
1. **传统方案**：大模型调用传统后端函数访问相关工具获得结果返回给用户，提问不直接到达大模型
2. **存在的问题**：
	1. 逻辑复杂且容易误判
	2. 调用函数的工具由后端创建，难度很大
3. **FunctionCalling方案**：
	1. 什么是FunctionCalling:
	    - 广义的FunctionCalling是指能够让大模型调用外部工具的一种技术实现：先向大模型提供可用函数的列表及说明，由大模型在对话过程中智能判断是否需要调用函数，并自动生成调用所需的参数，最终用文字返回符合约定格式的函数调用请求
	    - 狭义的FunctionCalling特指大模型提供商在模型内部与API层面做了支持的一种能力，它最早是由OpenAI引入：
		    - 在模型层面，模型提供商需对大模型进行特别优化，使其具备根据上下文正确选择合适函数、生成有效参数的能力（比如有监督微调、强化学习）
		    - 在API层面，模型提供商需额外开放对FunctionCalling的支持
	2. 基于提示词的FunctionCalling（广义）:
	     - 方法：通过系统提示词规定函数说明以及大模型的返回格式
	     - **存在问题**：
		       1. 输出格式不稳定。如调用指令中存在多余自然语言。
		       2. 容易出现幻觉，模型可能编造并不存在的函数名或参数。
		       3. 对开发者依赖度高，函数描述、调用指令格式、提示词逻辑完全由开发者设计
		       4. 上下文冗长，Token消耗高
	3. 基于API的FunctionCalling(狭义)
	    - 方法：
		      1. 用户发起提问
		      2. 后端第一次向大模型API发起请求，获取函数调用指令：后端向大模型API传入原始输入、函数描述和其它上下文信息，获取调用指令。函数描述包括函数名称、用途说明、参数结构等
		      3. 模型生成调用指令，模型会智能判断是否需要调用函数，选择合适的函数，并基于上下文自动生成结构化的调用指令
		      4. 后端解析调用指令，并执行实际的函数调用。后端接收到模型返回的调用指令后，解析调用指令，得到函数名称和参数，执行对应的方法并获取结果
		      5. 后端第二次向大模型API发起请求，将刚才的调用结果和其他上下文一起传给模型，生成最终的回复
		- 问题  ：
			  1. 后端应用适配不同的大模型时存在大量冗余开发
			  2. 可选模型有限
			  3. Token消耗高的问题也无法解决
## 二、MCP协议
### 要解决的问题
1. **工具接入的冗余开发问题**：
   AI应用接入他人开发的新工具需完整copy代码和函数描述，接入几个就要copy几次
2. **工具复用困难**：
   环境问题导致copy的代码不一定能跑；很多企业不提供源码；跨语言的代码copy了没用
### 什么是MCP（模型上下文协议）
- MCP是一个开放协议，用于标准化应用程序向大语言模型提供上下文的方式。借助MCP可以在LLM之上构建智能体和复杂工作流，并将模型与外部智能体相连接。
### MCP与Functioncalling的关系
- MCP本质上是对FunctionCalling进行了标准化而非替代关系，并在其基础上加入了Resources与Prompts。构建出一个即插即用的AI工具生态系统。
- 传统 Function Calling 中，工具列表是硬编码在请求里的；MCP 允许 Server 在连接时动态注册/注销工具，并且工具可以由第三方独立提供。
### MCP的核心架构
MCP遵循客户端-服务器架构，其中MCP Host——AI应用程序——与一个或多个MCP Server建立连接，MCP主机通过为每个MCP Server创建一个MCP Client来实现这一目标。每个MCP Client都与相对应的MCP Server保持专用的一对一连接。MCP架构的主要组成者是：
- **MCP Host**：协调和管理一个或多个MCP Server的人工智能应用程序
- **MCP Client**：一个组件用于维护与MCP服务器的连接，并从MCP服务器获取上下文，供MCP主机使用
- **MCP Server**:一个为MCP Client提供上下文的程序，其中有很多tools
### MCP的传输协议
#### Stdio传输
*本质上是进程间通信的一种形式，它最常用的底层机制就是管道*
1. 什么是Stdio:
    - **stdio(standard I/O)是进程的标准输入/输出接口**。每个进程启动时，操作系统会给它分配三个文件描述符：stdin/stdout/stderr
    - 程序里的printf、scanf、cin、cout、read、write都是通过这些接口和外界交换数据的
2. 什么是管道：
    - 管道是操作系统内部提供的一种进程间通信机制，它允许一个进程的输出直接作为另一个进程的输入，实现数据在两个进程之间的流动
3. 什么是stdio传输：
    - 通过标准输入和标准输出这两个数据流来传输数据、通过管道来连接两个进程的标准输入/输出接口，使得一个进程的输出直接传输给另一个进程输入，实现进程间数据传输。
    - stdio是接口，管道是连接接口的通道
#### HTTP+SSE传输（旧方案）
*用户通过HTTP POST向服务端发请求，服务端通过SSE通道返回响应结果*
- SSE（Sever-Sent Events服务器发送事件），是一种服务器单向推送数据给客户端的技术，基于HTTP协议
- 基本原理：
    - 客户端先向服务端发起一个HTTP请求
    - 服务端保持这个连接不断开，以text/event-stream作为响应类型，源源不断地往里写数据
    - 客户端收到数据后会触发响应地事件回调
- 与普通HTTP的核心差异
    - 支持服务端主动、流式的推送消息
- 服务端推送的必要性：MCP Server中的工具发生了更新，要主动地向MCP Client推送通知
#### Streamable HTTP传输（新方案）
- Streamable HTTP并不是一个标准协议名，而是一个通用描述，指的是基于HTTP协议的“可流式传输”技术。它的核心思想是：在一个HTTP连接里，服务端可以持续不断的发送数据给客户端，客户端边接收边处理，与“流”类似。与传统的HTTP请求响应“一次完成”不同，StreamableHTTP保持连接不关闭，数据分片持续传输。常见实现方式包括：
    - HTTP/1.1 长连接+分块传输编码
    - HTTP/2 流式数据
    - HTTP/3 QUIC流式传输
- 为什么HTTP+SSE要升级成Streamable HTTP？
    - **数据格式限制问题**：SSE的text/event-stream只支持文本格式；Streamable HTTP的Content-Type支持任意格式，更适合AI场景
    - **跨平台兼容问题**：SSE支持的客户端主要是浏览器和少量语言库；而StreamHTTP支持多种客户端
    - **性能问题**：SSE是基于HTTP/1.1的长连接，Streamable HTTP可以基于HTTP/2/3,支持多路复用和双向流。且HTTP/2/3的流控制和优先级机制使得高吞吐和低延迟成为可能。SSE消息只支持文本格式，StreamableHTTP可以采用更高性能更紧凑的其它编码方式。
# Agent
## 一、概述
### 概念
由大模型与可供大模型调用，可以完成任务，处理问题的工具组成的AI工具就叫作Agent
### Agent的种类
1. 按智能程度分类：
	- **简单反射型**：仅根据当前感知，直接匹配预设规则做出反应，不维护内部状态
	- **基于模型的反射型**：维护内部状态模型，能处理部分可观测环境，依据当前感知和内部模型做出决策
	- **基于目标型**：除状态模型外还拥有目标，会规划行动序列以达成目标
	- **基于效用型**：不仅追求目标，还要求最大化效用函数，在多目标和多方案中衡量得失，选出“最优”方案。
	- **学习型**：能从经验中改进性能，包含学习元件，可以自动更新规则、模型或效用
2. 按架构与实现范式（如今主流分法）
    - **ReAct Agent**:交替进行思考和行动，结合推理与工具调用。是目前最常见的LLM Agent模式
    - **Plan-and-Execute Agent**：先制定完整计划，再按步骤逐步执行，适合复杂、长周期任务
    - **多模态Agent**：能同时处理文本、图像、语音、视频等多种模态信息，与环境交互
    - **反思/自我批判型**：在输出行动后，对自己的行为进行检查、反思和纠错，形成反馈闭环
    - **多Agent协作系统**：多个专业Agent分工协作，有层级或对话式组织
## 二、Agent模式
### ReAct模式：
1. 运行流程：
	1. 提交任务
	2. 思考（Thought）
	3. 判断是否需要调用工具
	4. 需要调用的话就进行行动（Action）
	5. 观察程序执行结果(Observation)
	6. 继续思考，之后继续判断是否需要调用工具，如果需要，就循环上面的步骤
	7. 直到不需要再次调用工具，则给出最终答案(Final Answer)
2. ReAct模式如何实现：
   通过系统提示词，详细规定模型角色、运行规则、环境信息等，然后与用户提问一同发送给大模型

# LangGraph+MCP多智能体开发
## 一、LangChain
### 概述
1. **LangChain**是一个用于开发由大语言模型驱动的应用程序的框架，工具集合
2. LangChain简化了LLM应用程序生命周期的每个阶段：
	- **开发**：使用LangChain的开源构建模块，组件和第三方集成构建应用程序，使用LangGraph构建具有一流流式处理和人机协作的有状态代理。
	- **生产化**：使用LangSmith检查、监控和评估链，以便持续优化和部署
	- **部署**：将LangGraph应用程序转变为生产就绪的API和助手
3. **LangChain**的作用：
	   1. **连接外部数据**：可以让LLM从外部获取最新信息
	   2. **记忆**：帮Agent进行长短期记忆，使交互更个性化
	   3. **调用工具**：让Agent能够调用各种工具完成任务
	   4. **复杂任务拆解**：把用户的大需求拆成小步骤，逐步执行
4. LangChain的主要模块：
	- **链（Chains）**：把多个操作串起来
	- **工具（Tools）**：让Agent调用外部功能
	- **记忆（Memory）**:存对话历史或用户数据
	- **检索（RAG）**：从知识库或网络抓取最新信息
5. **局限**：
	- 不是开箱即用
	- 性能依赖LLM
	- 复杂场景还需要定制
### 接入模型的流程
1. 安装LangChain-community
2. 导入所需模块
3. 添加key
### 提示词模板
LangChain的提示词模引擎是其核心组件之一，主要用于构建和管理与大语言模型交互的提示词。
它的作用是**将用户输入的动态数据结构化地嵌入到预定义的模板中**，从而生成适合模型处理的提示词，提升模型输出的准确性和一致性。
### 输出格式化
输出格式化是LangChain的关键功能之一，用于将大语言模型的原始输出解析为结构化的数据格式，比如JSON、字典或自定义对象，既便于后续处理，也提高了输出的可读性和一致性
### 链式调用
在LangChain中，LLMChain是一个核心对象，用于实现基于大语言模型的链式调用。它将提示词模板、语言模型以及可选的输出解析器组合在一起，形成一个可重复执行的流程，简化了与LLM的交互
### 流式输出
是指用户在输入问题后，AI边生成边显示响应内容，逐步展示AI的“思考”过程
### LangChain的记忆系统
1. **为什么需要记忆系统**：可以使与AI的对话更具有连贯性
2. **作用**：
    - 上下文保持
    - 个性化交互
    - 长期学习
3. **ConversationBufferMemory（缓冲记忆）**：
    *完整存储所有对话历史，不做任何压缩或筛选
    - **优点**：保留全部细节，适合需要完整上下文的场景
    - **缺点**：Token消耗随对话时间线性增长，易超出模型的上下文窗口限制
4.  **ConversationBufferWindowsMemory**：
	1. **定义**：是LangChain提供的一种记忆机制，只存储最近的k条对话记录。k是指用户指定的对话窗口大小，表示保留多少轮对话。当新的对话发生时，旧的对话会被移除窗口，确保内存和Token在可控范围内。
	2. **特点**
	    - 滑动窗口机制
		- 固定内存占用
		- 可配置性
	3. **缺点**：
		- 丢失早期上下文
		- 不适合复杂场景
		- 窗口大小选择困难
5.  **ConversationBufferTokenMemory**：
	1. **定义**：是一种LangChain记忆机制，根据Token数量来限制存储的对话历史。用户可以指定一个最大Token数，系统会保留不超过该限制的最近对话内容。当新对话的加入导致Token总数超过限制时，最早的对话会被自动移除。
	2. **特点**：
		- Token驱动的截断
		- 模型适配性好
		- 动态窗口
		- 高校存储
	3. **缺点**：
		1. 依赖大语言模型
		2. 可能丢失早期信息
		3. 设置复杂性
		4. 不适合需要完整历史的场景
## 二、LangGraph
### 快速了解LangGraph
1. LangGraph是一个功能强大的大模型本地应用构建框架，但它并不是一个独立框架，而是LangChain框架的一个生态组件
